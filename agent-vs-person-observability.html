The Governance Gradient — four experiments in control

Four surfaces, one org, incrementally decreasing visibility. From fully native Agentforce to MCP-routed headless apps — how does Shield’s control plane hold up as you move away from the platform? Hypothesis: Agentforce wins by design because governance is baked in.

The customer challenge

The team gets the same question from customers every week. It started with one sentence from Rachel and grew into this POC.

“The real-life customer challenge that our team is being asked about all the time is what is being edited by an agent versus a person. How do you know if it’s an agent acting on behalf of a person versus the person did it? And how do you detect if an agent is doing something outside of the bounds of what it should be doing?” — Rachel Beard, 2026-05-22 onboarding chat

The three questions customers ask

Every variant of the conversation boils down to these.

  • Attribution — “Did the user edit that record, or did an agent edit it on their behalf?”
  • Containment — “How do we detect an agent doing something outside the scope of what it should be doing?”
  • Enforcement — “Can Shield & Transaction Security actually stop an agent, or just observe it after the fact?”

Why these are hard to answer today

The team has the surface knowledge but not the demo.

  • No one on the team has seen a live demo of Event Monitoring for Agentforce.
  • TSPs exist for Agentforce but no one can articulate a sharp use case that the agent’s permissions wouldn’t already cover.
  • MCP + A2A is the question of the moment — and the team admits they don’t have a good answer for it.
  • Customers don’t just want to know what we can see — they want to know where the blind spots are.

Hypothesis & the POC

What we set out to test, and the bench we built to test it.

The governance gradient hypothesis

Agentforce has a distinct advantage because governance is baked in.

  • Native Agentforce should deliver the highest level of control — full governance, full visibility, full TSP coverage — because the entire stack runs inside Salesforce.
  • As you move away from the platform toward external apps, governance should erode predictably, but the key question is: where does it actually break?
  • The token type an app uses — Connected App session token vs. ECA JWT — determines real-time Shield visibility more than the API path. An app can hit direct REST and still be invisible to Shield if it authenticates via ECA.

What I built to test it

A single org, four live consumer surfaces, one streaming monitor.

  • Live Event Monitor LWC on a FlexiPage in the org, polling RTEM BigObjects every 5s and tagging each card with a source-specific icon.
  • Agentforce agent built in Agent Builder with a topic that runs SOQL via tool calls.
  • Cursor IDE making sf data query calls via sf-cli.
  • sf-dashboard — a Node.js app using Direct REST, tested with both ECA JWT (invisible to Shield) and Connected App session token (fully visible). The auth method, not the API path, is the variable.
  • agentforce-today — a React app built from scratch that routes SOQL through the Hosted MCP gateway + Anthropic LLM for daily briefings.
  • Drafted a sample TSP Apex class to test whether enforcement would actually fire for each surface.
Spoiler — the gradient is real, with a twist at L3 Levels 1–2 maintain full Shield visibility. Level 3 is conditional — Connected App session token gives full visibility; ECA JWT gives none, even over direct REST. Level 4 (Hosted MCP) falls off the map entirely. The story isn’t just “Agentforce is better” — it’s that the token type is the governance variable customers are choosing without realising it.

L1Agentforce — full governance by design

The baseline. In-org agent, in-platform governance. Everything Shield offers is available here.

Step 01 How Agentforce connects
01 / 04

In-org agent built in Agent Builder

Agentforce agents live inside the org. They’re configured via Agent Builder with Topics, Actions, and Instructions. When the agent needs CRM data, it invokes a tool call that runs SOQL internally. No external API hop. The user’s session context (including identity) flows through the entire operation.

Agent runs in user’s session context
SOQL via internal tool call
No external REST endpoint involved
Step 02 What Shield sees
02 / 04

Full real-time visibility + agent attribution

Every SOQL generates an ApiEvent in real-time. Critically, the event includes BotId — the platform’s native identifier for the agent that ran the query. This is the cleanest attribution story: you can tell not just who but what agent performed the operation.

ApiEvent: real-time, full detail
BotId: agent-level attribution
LoginEvent: user session tracked
Step 03 Full governance stack
03 / 04

Agentforce-only controls — layers L2–L4 don’t have

Beyond TSPs and ApiEvent, Agentforce delivers an entire governance stack that external surfaces cannot replicate:

Einstein Trust Layer — prompt injection defense, toxicity filtering, PII masking at the LLM boundary
Guardrails on Subagents — declarative limits on what child agents can invoke
Human-in-the-loop — approval steps, confirmation prompts, escalation flows
Session tracing — full conversation replay with tool calls and reasoning steps
Event Logs — standard EventLogFile captures for audit
Enhanced Event Logs — richer agent-specific telemetry in Data Cloud

Plus full TSP coverage: scope policies to BotId != null for agent-only rules. Row limits, entity restrictions, time-of-day — all enforceable.

L3sf-dashboard — headless app, direct REST

Fully headless Node.js app. Direct REST — but governance depends on the token type, not the API path. Connected App = full visibility. ECA JWT = blind spot.

Step 01 How sf-dashboard connects
01 / 04

Headless Node app — token type is the critical variable

sf-dashboard is a local Node.js app that calls the org’s direct REST endpoint (/services/data/v66.0/query) — no MCP, no gateway. Observability is entirely determined by how it authenticates, not what API it calls. With a standard Connected App session token (00D…), queries are fully visible. With an ECA-issued JWT (eyJ…), they are not — even though the API call is identical.

Connected App → 00D session token → ✅ visible
ECA → eyJ JWT token → ❌ invisible
Same REST endpoint either way
Step 02 What Shield sees
02 / 04

What Shield sees — depends entirely on auth method

Connected App (session token): every query generates a real-time ApiEvent with full detail — user, SOQL, rows processed, ConnectedAppId, source IP. TSPs fire. Streaming monitor shows every call.

ECA (JWT token): zero ApiEvent records generated. Empirically verified — 200 records queried across the org, not one from an ECA-authenticated source. Activity only surfaces in the daily RestApi EventLogFile. The LoginEvent for the OAuth handshake is visible, but every subsequent query is a black box to real-time Shield.

Connected App: ApiEvent ✅ real-time
ECA JWT: ApiEvent ❌ zero records
ECA JWT: EventLogFile ✅ next day only
Step 03 TSP coverage
03 / 04

TSP enforcement — Connected App yes, ECA no

Connected App: TSPs fire on every query. Scope by ConnectedAppId for per-app governance — row limits, entity restrictions, time-of-day rules, block or alert. Fully enforceable.

ECA: TSPs evaluate on ApiEvent. Since ECA JWT sessions generate no ApiEvent records, policy conditions never trigger — not even to allow the query through. A row-limit rule that works perfectly for Connected App traffic is silently ignored for the same query from an ECA-authenticated app. No real-time control path exists today.

Connected App: TSP ✅ fully enforceable
ECA: TSP ❌ conditions never fire
ECA gap: platform roadmap item

L4agentforce-today — MCP routed, visibility gap

The cliff. Same user, same OAuth, same intent — routed through Hosted MCP. Governance breaks.

Step 01 How agentforce-today connects
01 / 04

React app built from scratch — routed through Hosted MCP

agentforce-today is a React app I built from scratch that uses the same PKCE OAuth flow as sf-dashboard. The difference: instead of hitting the org’s REST endpoint, it routes SOQL through Salesforce’s Hosted MCP gateway: api.salesforce.com/platform/mcp/v1/…. The user’s token authenticates them to the gateway, not through it.

React + Vite + Anthropic LLM
api.salesforce.com/platform/mcp/v1/...
User token authenticates TO gateway
Step 02 What Shield sees
02 / 04

LoginEvent only — no ApiEvent for queries

The OAuth login generates a LoginEvent — that part works. But the SOQL? It runs server-side as Salesforce’s internal cloud@… CloudIntegrationUser. No ApiEvent is generated for the user’s session. The query activity surfaces 24 hours later in ApiTotalUsage EventLogFile — aggregated, not real-time, and attributed to the system user.

LoginEvent: yes (real-time)
ApiEvent: NO — visibility gap
ApiTotalUsage: 24h lag, aggregated
Step 03 TSP coverage
03 / 04

No query-level TSP — only LoginEvent binary block

TSPs cannot enforce query-level policies because there is no ApiEvent to evaluate. The only enforcement option is a LoginEvent TSP that blocks the session entirely based on gateway IP or login pattern. It’s binary (allow or block the whole session), not granular. No row limits. No entity restrictions. No time rules.

No ApiEvent = no query TSP
LoginEvent TSP: binary block only
Gateway IPs: 3.234.75.13, 3.211.112.171

Why the gap exists — two API patterns, two trust boundaries

The visibility difference isn’t a bug. It’s a direct consequence of where the trust boundary lives in each pattern.

Direct REST (sf-dashboard)

User token authenticates AND authorises the operation at the data layer.

  • Calls the org’s instance URL directly: https://<org>.my.salesforce.com/services/data/v63.0/query.
  • OAuth bearer = the user; the SOQL runs as that user on the database.
  • Every query produces a real ApiEvent with username, IP, URI, method, rows, elapsed time, ConnectedAppId.
  • Visible in real time, evaluable by TSPs, audit trail is per-query and per-user.

Hosted MCP gateway (agentforce-today)

User token authenticates to the gateway only; execution happens as a system identity.

  • Calls api.salesforce.com/platform/mcp/v1/platform/sobject-all.
  • OAuth bearer authenticates the user to the gateway, but the SOQL runs server-side as cloud@… CloudIntegrationUser (Jetty/12.0.32, 155.226.x.x).
  • The user’s identity is not attached to the database operation in real time.
  • No ApiEvent. Audit only through next-day ApiTotalUsage EventLogFile, attributed via ConnectedAppId.

Where the per-user attribution actually breaks

The Hosted-MCP request chain has five real hops. Stage four is where user-attribution is lost — everything after it inherits the gateway’s service identity.

02

03
Gateway in

POST to api.salesforce.com/.../mcp with user token

ApiTotalUsage EventLogFile, 24h later, aggregated

What is always visible — regardless of pattern Every OAuth flow produces a LoginEvent with identity, source IP, login type (Remote Access 2.0), browser/UA, and timestamp. The gap is only at the query layer, not at authentication. That single fact is what makes a LoginEvent TSP the one real lever customers retain against the MCP path.
Why L1 has a distinct governance advantage Agentforce includes the Einstein Trust Layer — a security boundary between the agent and the LLM that handles data masking, prompt injection defense, toxicity filtering, and zero data retention. L2, L3, and L4 don’t get this layer at all. They’re just calling APIs directly; there’s no intermediary sanitizing inputs or vetting outputs.

Sample Transaction Security Policy

A concrete TSP Apex class drafted as part of the POC — it caps sf-cli queries at 1,000 rows. It fires for Agentforce and Cursor unconditionally. It fires for sf-dashboard only when authenticated via a Connected App session token — not when using an ECA JWT. It does not fire for agentforce-today because no ApiEvent is generated there.

CursorRowLimitPolicy.cls
// TSP Apex class — evaluates ApiEvent in real time. // Triggered: "API event with row count over threshold from sfdx-like client". public class CursorRowLimitPolicy implements TxnSecurity.EventCondition { public Boolean evaluate(SObject event) { Map<String,String> d = ((ApiEvent)event).getData(); Integer rows = Integer.valueOf(d.get('RowsProcessed') ?? '0'); String client = d.get('Client') ?? ''; String user = d.get('Username') ?? ''; // Cap any sfdx-driven session at 1,000 rows per call. return rows > 1000 && client.containsIgnoreCase('sfdx'); } }
When this fires, the TSP can block the query, end the session, fire an in-app notification, send an email to the security team, or post to a SIEM webhook — configured per policy in Setup. The Apex class is just the condition; the action is declarative.

Conditions library — what else you can express

Each condition below is a one-liner against an ApiEvent field. They compose with && / || in the evaluate() method.

Condition Example
Too many rows returned RowsProcessed > 1000
Specific user Username = 'storm.3d…'
Specific client Client LIKE '%sfdx%'
Time of day EventDate outside business hours
Specific sensitive object QueriedEntities CONTAINS 'Lead'
Agent-originated only BotId != null (Agentforce only)
Headless ECA only ConnectedAppId IN (set of headless ECAs)

Where this TSP fires across the four surfaces

Surface ApiEvent produced? Sample TSP fires? If you need enforcement against this surface…
Agentforce Yes Yes — with BotId != null filter Use BotId as the discriminator; you can scope per-bot policies.
Cursor (sf-cli) Yes Yes Client LIKE '%sfdx%' matches; the sample above is exactly this case.
sf-dashboard (Direct REST) Yes Yes — with ConnectedAppId filter Scope by ECA ConnectedAppId so the policy is app-specific.

The architectural takeaway in one line A TSP can control who gets a token but not what that token is used to query once the MCP gateway is in the path. The enforcement boundary ends at authentication for the Hosted-MCP surface.

Shield control coverage against MCP-mediated agents

Shield was designed around a direct-API model: every operation surfaces as an ApiEvent, every Shield control hangs off that. The Hosted MCP gateway introduces a new trust boundary that sits above that control plane.

Shield control Effective vs MCP-mediated queries? Why
Event Monitoring (RTEM) No No ApiEvent generated for MCP-routed queries.
Transaction Security — query-level No No event to evaluate; only login-level TSP retains leverage.
Field Audit Trail No Tracks writes only; MCP read queries leave no trail there.
Platform Encryption No Encrypts data at rest; an authenticated MCP session still gets plaintext for permitted fields.
LoginEvent TSP Yes — binary block at auth Only Shield control with teeth against this surface — but all-or-nothing, not granular.
EventLogFile (daily audit) Yes — 24h lag Auditable after the fact via ApiTotalUsage; cannot drive real-time response.
The architecturally significant finding: every Shield control that could restrict data access (as opposed to data storage) operates downstream of ApiEvent. Since MCP bypasses that event surface, none of them have visibility into query operations. The only Shield control with real teeth against MCP-based agents is a LoginEvent TSP — and that’s a binary block, not granular data-level governance.

Apply this to a Shield customer — compensating controls

Four practical controls a Shield customer can deploy today, in order of leverage, until Salesforce closes the real-time MCP gap (likely a dedicated McpRequestEvent).

01
LoginEvent TSP on the MCP gateway IP range

Detect login type API with SourceIp in the Hosted-MCP gateway range (3.234.75.13, 3.211.112.171). Block the session or fire an alert at authentication. Binary, immediate, the only real-time lever you have on this surface.

Effort: Low · Latency: Real-time
02
Nightly ApiTotalUsage pipeline into the SIEM

Pull EventLogFile for ApiTotalUsage and Login daily, join on ConnectedAppId for user attribution, land it in your SIEM. Drives detection rules with a one-day lag — better than nothing, auditable, ECA-traceable.

Effort: Medium · Latency: ~24 hours
03
Network egress monitoring at the MCP-gateway layer

If the MCP-connected app runs in your VPC or behind a forward proxy, log the outbound calls to api.salesforce.com/platform/mcp/v1/… at the network layer. You get request/response metadata Salesforce will not give you in ApiEvent.

Effort: Medium · Latency: Real-time
04
Shield Data Detect on sensitive fields

Layer-of-last-resort: tag PII and revenue-sensitive fields with Data Detect policies that fire on any access. This operates at the data classification layer and is independent of which API path the access came from.

Effort: High · Latency: Real-time
Key takeaway Agentforce gives real-time, per-operation visibility in Shield today. Headless MCP via Salesforce’s own Hosted MCP server gives daily aggregated logs only — individual query-level audit trails do not exist yet. For customers with compliance or security-monitoring requirements, this is a gap Salesforce will need to close before Hosted MCP is production-ready in regulated environments.
