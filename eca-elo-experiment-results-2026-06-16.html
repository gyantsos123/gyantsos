Agent Observability for MCP / ECA Apps

What Shield can and can't see today — a capability assessment for Product & Security Architects · June 2026

Purpose & audience: This briefing is for product managers and security architects. It documents, with live evidence, exactly what Shield Event Monitoring captures for an AI agent app that reaches Salesforce data through the Hosted MCP gateway using an External Client App (ECA) — and where the gaps are. Section 7 translates each gap into a concrete product change.
Executive summary: For MCP/ECA-routed agents, Shield can confirm the app authenticated and can show the volume of data operations — but it cannot attribute those operations to a specific app, cannot show what was queried, and cannot enforce against the activity in real time. The data is timely (hourly/real-time); the gaps are attribution, query detail, and enforcement, all of which require product changes in the MCP gateway's eventing — not customer configuration.

1. Timeliness is not the problem

With hourly Event Monitoring enabled, an agent's activity is observable the same day. Three access tiers, fastest first:

TierLatencyAvailability (test org)
Real-time events (LoginEvent only for MCP)~15 minLogin only — see note
Hourly log files~1–3 hrsPresent (~72 files/day)
Daily log file rollup~24 hrs+Lags ~1 day

Note: in the real-time tier, only the LoginEvent appears for an MCP app. ApiEvent is real-time for other surfaces (e.g. Agentforce, direct REST) but is never generated for MCP-routed queries — so the query activity is absent here entirely, not just delayed. Takeaway for Product: latency is acceptable; the problems below are about content and enforceability, not freshness.

2. What signal each tier carries for an MCP/ECA app

RTEM and Event Log Objects read the same events (ApiEvent, LoginEvent) — same fields, different delivery. Log Files (ApiTotalUsage, RestApi, Login) are a separate dataset. This is where each signal lives:

Why the gateway-user, MCP-flag, client, and app-id rows are dashes for RTEM/ELO: those are RestApi/ApiTotalUsage file fields, not fields on the ApiEvent/LoginEvent objects. Verified against 6/15 ELO data — ApiEvent returned 0 gateway (cloud@…) rows and LoginEvent returned 0 oauthc2ctokenexchange rows — so no queryable object carries the MCP query activity at all.

3. The attribution gap GAP

Once a query enters the Hosted MCP gateway, it executes as Salesforce's internal CloudIntegrationUser. Every field that would identify the originating app — CONNECTED_APP_NAME and CONNECTED_APP_ID — is blank. The login is named (LoginEvent.Application); the queries are not. A customer can see that MCP traffic occurred, but not which agent app produced it. With more than one MCP app in an org, the activity is indistinguishable.

4. The enforcement gap GAP

A Transaction Security Policy can only act on a real-time event. For MCP the only real-time event is the LoginEvent, so the single available control is blocking the login — an all-or-nothing denial. Because no ApiEvent is emitted for the query, there is no query-level enforcement: no row caps, entity restrictions, rate limits, or time-of-day rules. You can deny the session; you cannot govern what the agent does inside it.

5. The best a customer can do today (workaround & its limit)

Isolate MCP volume (in RestApi): USER_TYPE = CloudIntegrationUser OR CLIENT_NAME = agentic-api-service; exclude Data Cloud and CLI rows.
Attach a name: correlate to the LoginEvent.Application login.

Limit: this isolates "traffic the gateway executed," which is a category, not an app identity — and it only works cleanly when a single MCP app is active. It is not a substitute for native attribution.

7. What would close the gap FOR PRODUCT

None of these are customer-configurable — they require changes in how the MCP gateway emits telemetry. In priority order:

#Gap todayWhy it mattersWhat would close it
1MCP queries emit no real-time event (no ApiEvent)No live visibility; TSPs can't evaluate query activityEmit a real-time event for MCP-routed data operations (extend ApiEvent or a new MCP event)
2MCP rows carry no originating app identity (runs as CloudIntegrationUser; app id/name blank)Activity can't be attributed to a specific agent app; multiple apps are indistinguishablePropagate the originating ECA's client id / app name onto the gateway's events and EventLogFile rows
3Enforcement is login-only (binary block)No row limits, entity scoping, or rate limits on agent queriesExtend Transaction Security to the MCP query path (depends on #1)
4No query detail — SOQL text blank, entity partialCan't tell what data the agent accessedCapture query text and full entity in MCP-path logging

The one-line ask for Product: make MCP-gateway data operations emit a real-time, app-attributed event — that single change unlocks attribution (#2), enforcement (#3), and query detail (#4) at once.
